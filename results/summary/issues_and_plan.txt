Project comparison: local implementation vs. "statgenomics - kehinde"
Date: 2026-04-26

Purpose
This file summarizes the main issues identified while comparing the two implementations
and gives a prioritized plan for producing defensible final results.

Key conclusion
The two result sets should not be expected to match directly because they are not analyzing
the same RNA/ATAC cell pair. In addition, Kehinde's implementation appears to use different
RNA inputs than the assignment requires and likely swaps ATAC sample labels. The local RNA
analysis is closer to the project specification, but the ATAC workflow still needs method
improvements before it should be treated as the final version.

Current status update (after pipeline updates through 2026-04-27)
- RNA inputs and filtering are now standardized through shared helper code and ScriptSeq-only manifests.
- RNA outputs, clustering, and summary files are internally consistent and currently represent the strongest part of the project.
- ATAC scripts are now BAM-ready and will automatically switch to BAM counting when the expected BAM files are present.
- While BAMs are still unavailable in the current run, ATAC remains in bigBed score fallback mode.
- RNA-ATAC integration outputs now exist, but they inherit the current ATAC limitations and must be treated as exploratory.
- Additional exploratory ATAC figures now include TSS-distance and annotation-composition plots for presentation.
- Remaining major limitation: ATAC still uses bigBed score-derived values with DESeq2 in the current results, which is not ideal for inferential statistics.

======================================================================
ISSUES IDENTIFIED
======================================================================

1. The two implementations analyze different cell pairs

Local implementation:
- RNA scripts analyze CMP vs Erythroblast
- ATAC script analyzes CMP vs Erythroblast

Kehinde implementation:
- RNA script analyzes HSC vs CMP
- ATAC script analyzes HSC vs CMP

Why this matters:
- Differential expression and differential accessibility are pair-specific.
- DEG counts, top genes, GO terms, PCA separation, and volcano plots are expected to differ.
- A mismatch in results is not evidence by itself that the local implementation is wrong.


2. Kehinde's RNA input selection does not match the assignment specification

Assignment requirement:
- Use ScriptSeq RNA-seq files only

Observed in encode ids.txt:
- HSC ScriptSeq: ENCFF247FEJ and ENCFF064MKY
- HSC TotalScript: ENCFF040EGE and ENCFF250AWY

Observed in Kehinde's script:
- Uses ENCFF040EGE and ENCFF250AWY for HSC

Why this matters:
- Kehinde is using TotalScript for HSC instead of ScriptSeq.
- That makes his RNA results not directly comparable to the local implementation.
- On this point, the local implementation follows the assignment more closely.


3. Kehinde's ATAC sample labels appear to be reversed

Observed in encode ids.txt:
- CMP ATAC: ENCFF832UUS and ENCFF343PTQ
- HSC ATAC: ENCFF662DYG and ENCFF255IVU

Observed in Kehinde's script:
- HSC_atac <- ENCFF832UUS.bigBed
- CMP_atac <- ENCFF662DYG.bigBed

Why this matters:
- The labels for HSC and CMP appear swapped.
- This would reverse the biological interpretation of regions "open in HSC" vs "open in CMP".
- ATAC conclusions from that script should be treated cautiously.


4. Local RNA analysis appears broadly reasonable

Observed local strengths:
- Uses ScriptSeq input files listed in encode ids.txt
- Uses both biological replicates
- Runs both DESeq2 and limma-voom
- Applies low-count filtering
- Produces DEG tables, GO enrichment, clustering, and method comparison summaries

Interpretation:
- No major conceptual problem was found in the local RNA workflow.
- The disagreement with Kehinde's results is mainly explained by different cell pairs and
  different input choices, not by an obvious RNA bug in the local code.


5. Local ATAC workflow is exploratory in the current run, not the strongest final method

Observed local method:
- Imports processed bigBed peak files
- Builds a consensus peak set
- Uses bigBed peak scores as DESeq2 input

Why this matters:
- DESeq2 is designed for count-based data.
- bigBed "score" values are not the same as read counts from aligned BAM files.
- The current local ATAC workflow can be useful for exploratory analysis, but it is weaker
  than a proper differential accessibility pipeline based on read counts per peak.

Consequence:
- ATAC results may be directionally informative, but they should be described carefully.
- TSS-distance and annotation plots are useful for presentation now, but they do not fix the underlying quantification issue.
- For a final project version, BAM-based peak counting would be more defensible.


6. Local ATAC peak-consensus logic is now mostly aligned, but interpretation remains limited by fallback quantification

Pairwise ATAC and all-cell ATAC clustering scripts:
- Now use the same union-based replicate consensus strategy
- Still differ in purpose: pairwise testing vs all-cell structure exploration

Why this matters:
- The previous union/intersection inconsistency has been reduced.
- The bigger remaining issue is that both ATAC branches still depend on fallback score-based quantification in the current run.


7. RNA identifier handling should be made more explicit and robust (moderate priority)

Observed:
- Raw RNA files include many low or zero rows near the top with non-Ensembl-looking gene_id values.
- Expressed rows used in downstream results appear to include standard ENSMUSG identifiers.
- Current GO code assumes the tested IDs are Ensembl gene IDs after filtering.

Why this matters:
- The current results look biologically plausible, but the preprocessing should explicitly keep
  valid gene-level ENSMUSG identifiers before annotation and GO.
- This would reduce ambiguity and make the workflow easier to defend in the report.

Practical note:
- This is a quality/reproducibility improvement, not the most likely source of large result discrepancies.


8. RNA and ATAC comparisons should be judged by assignment goals, not by exact agreement

Why this matters:
- The assignment explicitly asks to compare DESeq2 and limma-voom for the first pair.
- It does not require different students or different cell pairs to have identical outputs.
- The correct test is whether the workflow is consistent, reproducible, biologically sensible,
  and aligned with the assigned data and methods.


======================================================================
PRIORITIZED PLAN TO SOLVE THE ISSUES
======================================================================

Phase 1 (must fix). Lock the correct inputs

1. Confirm the assigned cell pairs for the team and document the order of analysis.
2. Use encode ids.txt as the source of truth for all sample identities.
3. Restrict RNA analysis to ScriptSeq files only.
4. Reconfirm ATAC sample labels before any downstream reruns.

Expected outcome:
- All analyses are based on the correct files and the same interpretation of sample identity.


Phase 2 (must fix). Standardize the RNA workflow

1. Build one clean RNA pipeline template for any cell pair.
2. Read expected_count from the ScriptSeq files for both replicates.
3. Keep only valid gene-level Ensembl IDs for downstream annotation.
4. Apply a documented low-expression filter.
5. Run both DESeq2 and limma-voom on the first assigned pair.
6. Compare the methods using:
- number of significant genes
- fold-change agreement
- overlap of DE genes
- biological interpretability of GO results
- replicate clustering and PCA separation
7. Choose the stronger RNA method for the second pair and later pairs, as required by the assignment.

Expected outcome:
- A consistent and defensible RNA analysis pipeline that directly answers Questions 1 to 4.


Phase 3 (must fix for strongest claims). Strengthen the ATAC workflow

Best-practice plan:
1. Use BAM files if available.
2. Define a consensus peak set across replicates.
3. Count reads per peak per sample.
4. Run DESeq2 or edgeR on peak count matrices.
5. Annotate significant DARs to nearby genes.
6. Perform GO enrichment on genes linked to DARs.

Fallback plan if BAM processing is not feasible:
1. Keep the bigBed-based workflow.
2. Clearly state in the report that peak scores are being used as an approximation.
3. Avoid overstating statistical certainty.
4. Use only exploratory visuals from ATAC:
- replicate correlation
- PCA / clustering
- annotation composition
- distance-to-TSS summaries
- a small number of concordant example genes

Expected outcome:
- A more defensible approach for Questions 5 to 8, with limitations clearly stated if BAM-based
  counting cannot be completed in time.


Phase 4 (largely implemented). Make ATAC definitions consistent across analyses

1. Use the same union-based replicate peak strategy in pairwise ATAC and all-cell ATAC clustering.
2. Keep that rule documented in the methods section and script comments.
3. Re-evaluate peak-definition strategy only if BAM-based reruns reveal clear overexpansion of the consensus set.

Expected outcome:
- Pairwise accessibility analysis and all-cell ATAC clustering become directly comparable.


Phase 5 (implemented, but exploratory until BAM rerun). Integrate RNA and ATAC after both are stable

1. Finalize DE gene results first.
2. Finalize DAR results second.
3. Link DARs to nearby genes.
4. Compare directionality:
- genes upregulated in one cell type
- nearby regions more accessible in the same cell type
5. Perform GO enrichment on overlapping RNA/ATAC-supported gene sets.
6. Prefer promoter/proximal-linked integrated genes for presentation before using distal-linked genes.

Expected outcome:
- A clean answer to the question of how chromatin accessibility relates to nearby gene expression.
- Under the current fallback run, this answer should be framed as a candidate/regulatory hypothesis set rather than a definitive mechanistic result.


Phase 6 (must fix). Prepare the report around correctness and limitations

1. For each analyzed pair, report:
- input files used
- filtering rules
- statistical method
- significance thresholds
- DEG and DAR counts
- GO results
- clustering interpretation
- integration summary
2. Include a method-comparison section for DESeq2 vs limma-voom on the first pair.
3. Include a limitations section, especially if the ATAC analysis remains bigBed-based.
4. Make sure conclusions are phrased pair-by-pair rather than assuming all pairs behave the same.

Expected outcome:
- A report that is transparent, reproducible, and aligned with the assignment rubric.

Defer (good improvements, not blockers)

2. Additional sensitivity analyses (e.g., alternate peak thresholds).

Remove from immediate concern
1. Expectation that your outputs should numerically match Kehinde's pairwise outputs.
2. Any direct "who is right" comparison without first matching pair and inputs.

RECOMMENDED NEXT ACTIONS
======================================================================

1. Keep RNA figures and summaries as the primary evidence base for the current presentation.
2. Use ATAC replicate/PCA/annotation/TSS-distance plots only as exploratory support until BAM mode is active.
3. Rerun pairwise ATAC, Q8 ATAC clustering, ATAC post-processing, and RNA-ATAC integration after BAM files are available.
4. After the BAM rerun, replace any fallback-based ATAC counts, GO terms, and integration claims in storyline/evidence files.
5. For the final report, prioritize promoter/proximal concordant RNA-ATAC examples over large global ATAC counts.


Bottom line
The local implementation is now in a good position for RNA-seq and for presentation-ready exploratory ATAC visuals.
The main unresolved issue is not code organization anymore; it is that the current ATAC biological claims are still limited by score-based fallback quantification.
Once BAM mode is active, the ATAC and integration layers should be rerun and promoted from exploratory support to evidence-bearing results only if the new outputs remain biologically coherent.
